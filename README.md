# Интеграционная платформа и аналитический модуль на Laravel 11

Данный репозиторий представляет собой комплексное решение, объединяющее бэкенд-инфраструктуру на Laravel 11 с базой данных SQLite и оптимизированные фронтенд-компоненты на чистом JavaScript (Vanilla JS). Все три задачи интегрированы в единую архитектуру проекта.

## 🛠️ Запуск, развертывание и тестирование проекта

### Требования к окружению
* PHP >= 8.2 (с расширением `pdo_sqlite`)
* Composer

### Инструкция по развертыванию

1. Установите зависимости:
   ```bash
   composer install
   ```
2. Скопируйте файл окружения и настройте подключение к SQLite:
   ```bash
   cp .env.example .env
   ```
   В `.env` укажите:
   ```ini
   DB_CONNECTION=sqlite
   CACHE_STORE=file
   ```
3. Запустите миграции (с полной очисткой базы от старых таблиц):
   ```bash
   php artisan migrate:fresh
   ```
4. Добавьте администратора в БД через сидер:
    ```bash
    php artisan db:seed --class=AdminUserSeeder
    ```

## Использование

*   **Ручной запуск команды сбора данных**:
    ```bash
    php artisan crypto:fetch-prices
    ```
*   **Локальная имитация работы планировщика (каждые 5 минут)**:
    ```bash
    php artisan schedule:work
    ```
*   **Запуск API сервера**:
    ```bash
    php artisan serve
    ```

## 📈 Спецификация решений по задачам

### 1. Автоматический сбор данных (Binance API)

#### Архитектурные особенности:
* **Изоляция сетевого слоя**: Вся логика запросов инкапсулирована в `BinanceService`.
* **Современный синтаксис**: Команда использует нативные PHP-атрибуты (`#[Signature]`, `#[Description]`).

* **Трансформация JSON**: Выдача данных из БД клиентам изолирована через `CryptoPriceResource` для маскировки внутренней структуры таблиц.

#### Реализованные тест-кейсы:

1. **`FetchCryptoPricesTest` (Тестирование консольной команды)**
   * **Что делает:** Проверяет полный цикл работы команды `crypto:fetch-prices`.

2. **`CryptoPriceApiTest` (Тестирование API-эндпоинта)**

3. **`PageVisitTrackerTest` (Тестирование счетчика посещений)**
   * **Что делает:** Проверяет эндпоинт `/api/visit` на корректность приема аналитики и интеграцию с GeoIP.

Дополнительно тестирует сценарий падения внешнего GeoIP-сервиса (500 Error) и проверяет, что Laravel-приложение корректно обрабатывает исключение, записывая дефолтный статус города без падения в рантайме.

📂 **`/resources/js/solution.js`**
---

#### JavaScript: Анализ и обоснование алгоритма фильтрации полей

При реализации фронтенд-части задания ключевой приоритет отдавался надежности алгоритма, чистоте кодовой базы (Vanilla JS) и производительности на стороне клиента.

#### Выбранный алгоритм: Извлечение числовых маркеров и строгое сравнение (`parseInt`)

Алгоритм работает по принципу нормализации данных: из строки имени поля (`name`) с помощью операции `replace(/\D/g, '')` удаляются все нецифровые символы, после чего полученная строка принудительно приводится к целочисленному типу `Number` через `parseInt(..., 10)`. Сравнение с выбранным типом происходит строго по значению примитивов.

#### Почему выбран именно этот вариант:
1. **100% защита от пересечения индексов (Баг «1 и 12»)**: Текстовый поиск считает, что элемент `name="text-12"` содержит в себе единицу, из-за чего ошибочно отображает его при выборе Типа `1`. Численное сравнение (`12 === 1`) гарантирует строгую изоляцию групп полей.
2. **Отказоустойчивость (Fault Tolerance)**: Внедрена защита от полей без цифровых индексов (например, служебные поля `name="submit"` или `name="_token"`). Алгоритм подменяет пустой результат на безопасный `0`, предотвращая появление `NaN` и ложные скрытия системных элементов формы.
3. **Сохранение целостности интерфейса**: Скрипт управляет видимостью не самих инпутов, а их ближайших логических контейнеров (`p`, `label`, `div`) через нативный метод `Element.closest()`. Это гарантирует, что сетка верстки, отступы и подписи к полям не «ломаются» и не сдвигаются.

---

#### Анализ алгоритмов-аналогов и причины отказа от них

При прочих равных условиях были рассмотрены три альтернативных инженерных подхода. Ниже приведено обоснование, почему они уступают выбранному решению.

#### Аналог №1: Текстовый поиск подстроки (`String.prototype.includes` / `indexOf`)
* **Суть алгоритма**: Проверка вхождения строки значения селекта в строку имени поля: `fieldName.includes(selectedValue)`.
* **Причина отказа**: **Бизнес-логика будет сломана.** Как было описано выше, данный алгоритм не способен отличить «Поле 1» от «Поля 12» или «Поля 21». При масштабировании формы (если типов станет больше 9) интерфейс начнет работать некорректно, отображая лишние элементы.

#### Аналог №2: Регулярные выражения со строгими границами слова (`RegExp` с метасимволом `\b`)
* **Суть алгоритма**: Динамическое создание регулярного выражения вида `new RegExp('\\b' + value + '\\b')` для поиска точного совпадения цифры.
* **Причина отказа**: **Высокая хрупкость и зависимость от верстки.** Символ дефиса (`-`) или подчеркивания (`_`) в названиях полей (например, `text-1` или `type_1`) интерпретируется движками JavaScript по-разному в контексте границ слова `\b`. Это приводит к «миганию» интерфейса на разных браузерах. Кроме того, конструирование объектов `RegExp` внутри циклов создает лишнюю и необоснованную нагрузку на сборщик мусора (Garbage Collector).

#### Аналог №3: Использование библиотеки jQuery и селекторов атрибутов
* **Суть алгоритма**: Использование встроенных CSS-выборок: `$('input[name*="' + value + '"]').show()`.
* **Причина отказа**: **Необоснованное усложнение архитектуры (Overengineering).** Подключение сторонней тяжелой библиотеки ради 40 строк логики — грубый антипаттерн в современной фронтенд-разработке. Современный стандарт DOM API предоставляет нативные, более быстрые и легковесные инструменты (`querySelectorAll`, `closest`, `addEventListener`), работающие без внешних зависимостей. Это критически важно при разработке виджетов для CRM-систем (таких как amoCRM), где скорость загрузки скрипта влияет на UX пользователя.

### 3. Аналитический модуль «Счетчик посещений»

Скрипт дистрибуции (сборщик метрик), который можно скопировать и **подключить на абсолютно любой сторонний сайт в интернете**, физически лежит по следующему пути:
📂 **`/public/tracker.js`**

#### Как это работает (HTML-сниппет для вставки на сайт):
Пользователю/клиенту достаточно вставить следующий асинхронный код в тело HTML-страницы любого своего ресурса:

```html
<!-- Counter Dynamic Tracker -->
<script type="text/javascript">
   (function(m,e,t,r,i,k,a){
       m[i]=m[i]||function(){(m[i].a=m[i].a||[]).push(arguments)};
       m[i].l=1*new Date();
       k=e.createElement(t),a=e.getElementsByTagName(t),k.async=1,k.src=r,a.parentNode.insertBefore(k,a)
   })(window, document, "script", "http://127.0.0.1:8000/tracker.js", "counter");

   counter({ id: 1 });
</script>
<!-- /Counter Dynamic Tracker -->
```
*(При деплое на реальный хостинг адрес `http://127.0.0.1:8000/tracker.js` меняется на публичный домен).*

#### Важнейшие архитектурные решения счетчика:
1. **Определение GeoIP на стороне бэкенда**: JavaScript на клиенте принципиально не может узнать реальный IP-адрес и город пользователя без использования сторонних платных сервисов. В данном решении реализован правильный коммерческий подход: `tracker.js` собирает только метрики устройства, а IP-адрес и определение города происходят строго на стороне Laravel при получении POST-запроса через бесплатный, быстрый и отказоустойчивый сервис `ip-api.com` (с ограничением таймаута в 3 секунды, чтобы не вешать входящий поток запросов).
2. **Динамический URL (Zero Configuration)**: Скрипт `tracker.js` автоматически вычисляет адрес главного сервера Laravel через парсинг `document.currentScript.src`. Его **не нужно переписывать вручную** при переносе проекта с локальной машины (`localhost`) на продакшен-хостинг.
3. **Использование `keepalive: true`**: Статистика отправляется через встроенный Fetch API с флагом `keepalive`. Это позволяет браузеру гарантированно завершить отправку POST-запроса в бэкграунде, даже если посетитель кликнул по внешней ссылке и мгновенно закрыл/обновил текущую страницу.
4. **Кросс-доменная безопасность (CORS)**: Конфигурация API Laravel открыта для внешнего мира (`allowed_origins => ['*']`), что позволяет беспрепятственно собирать аналитику с любых внешних доменов.
5. **Точность визуализации по ТЗ**: На закрытой панели администратора (`/dashboard`) графики построены с помощью **Chart.js**. Согласно условию технического задания, линейный график посещений по часам отображает **количество уникальных посещений по оси X**, а **время — по оси Y**. Для этого в конфигурацию графика внедрено инвертирование осей (`indexAxis: 'y'`), формирующее строгую горизонтальную диаграмму. Группировка данных и подсчет уникальных IP за каждый час производятся на уровне оптимизированного запроса к SQLite через `strftime`.

#### Как протестировать локально:
1. Убедитесь, что сервер запущен (`php artisan serve`).
2. Откройте в браузере демонстрационную страницу: `http://127.0.0.1:8000/test-counter.html`. Она автоматически загрузит скрипт и отправит метрики в базу.
3. Перейдите в панель управления: `http://127.0.0.1:8000/`, введите логин/пароль админа и изучайте интерактивные графики распределения трафика.
